Joins — inner, left, right, full, semi, anti, cross, broadcast

Multiple DataFrames ko combine karna. PySpark mein join SQL se thoda alag hota hai — syntax, null handling, aur performance sab samajhna zaroori hai.

inner join left join right join full outer left semi left anti cross join broadcast join self join

Key Concepts — Joins

  • inner — Only matching rows from both sides. Default join type.
  • left / left_outer — All rows from left + matching from right. NULL if no match.
  • right / right_outer — All rows from right + matching from left.
  • full / full_outer — All rows from both sides. NULL where no match.
  • left_semi — Only left rows WHERE match exists. Like SQL EXISTS. No right columns returned.
  • left_anti — Only left rows WHERE NO match exists. Like SQL NOT EXISTS.
  • cross — Cartesian product. Every row × every row. Use carefully!
  • broadcast() — Send small DF to all executors. Avoids shuffle. Use when one DF < 10MB.
  • NULL in join — NULL != NULL, so NULL keys never match in joins.

01What join strategies does Spark use? When is each chosen?▶
💡 Solution
# Spark chooses join strategy based on data size: # 1. BROADCAST HASH JOIN (fastest) # - One side < autoBroadcastJoinThreshold (default 10MB) # - Small table sent to ALL executors # - No shuffle needed! result = emp_df.join(broadcast(dept_df), "dept_id") result.explain() # BroadcastHashJoin # 2. SORT MERGE JOIN (default for large tables) # - Both sides sorted on join key, then merged # - Requires shuffle — both sides repartitioned # - Good for large + large joins spark.conf.set("spark.sql.autoBroadcastJoinThreshold", "-1") # disable broadcast result = emp_df.join(dept_df, "dept_id") result.explain() # SortMergeJoin # 3. SHUFFLE HASH JOIN # - One side hashed, other probes the hash table # - Used when one side is small but > broadcast threshold # 4. CARTESIAN (CROSS) JOIN # - No join condition — full cross product # - Very expensive, avoid unless needed # Force a strategy: result = emp_df.hint("broadcast").join(dept_df, "dept_id") result = emp_df.hint("merge").join(dept_df, "dept_id")
02left_semi vs inner join — key differences?▶
💡 Solution
# INNER JOIN: # - Returns columns from BOTH tables # - If right has multiple matches → LEFT row appears MULTIPLE times (duplicates!) inner = emp_df.join(orders_df, "emp_id", "inner") print(inner.count()) # > 10 because emp 1 has 2 orders # LEFT SEMI JOIN: # - Returns ONLY LEFT columns # - Each left row appears AT MOST ONCE (no duplicates!) # - Faster than inner join (no data from right side fetched) semi = emp_df.join(orders_df, "emp_id", "left_semi") print(semi.count()) # ≤ 10 — each emp appears once # Use left_semi when: # - You only need left table columns # - Just checking IF match exists (filter/existence check) # - Avoiding duplicates from one-to-many relationships
03Why does NULL in join key cause rows to be dropped?▶
💡 Solution
# NULL != NULL in SQL/PySpark # So NULL join keys NEVER match — rows are silently dropped! # Example: Meena has salary=NULL, not dept_id=NULL # But if we had NULL in join key: null_emp = emp_df.withColumn("dept_id", F.when(F.col("name") == "Pooja", F.lit(None)) .otherwise(F.col("dept_id")).cast("integer")) result = null_emp.join(dept_df, "dept_id", "inner") result.show() # Pooja is MISSING! NULL dept_id didn't match # Fix: Use LEFT join so Pooja shows with NULL dept info result = null_emp.join(dept_df, "dept_id", "left") # Or: Replace NULL key before join null_emp.fillna({"dept_id": -1}) \ .join(dept_df, "dept_id", "left") \ .show()
04Real DE scenario: Reconciliation — find missing records between source and target.▶
💡 Solution
# Source: what should be in target source = emp_df # Target: what actually got loaded (missing some) target = emp_df.filter(F.col("dept_id") != 30) # Missing in target missing = source.join( target.select("emp_id"), on="emp_id", how="left_anti" ) print(f"Missing records: {missing.count()}") missing.show() # Extra in target (shouldn't be there) extra = target.join( source.select("emp_id"), on="emp_id", how="left_anti" ) print(f"Extra records: {extra.count()}") # Mismatch in values (same key, different values) mismatch = source.alias("s").join( target.alias("t"), on="emp_id", how="inner" ).filter( F.col("s.salary") != F.col("t.salary") ) print(f"Value mismatches: {mismatch.count()}")
05Optimize a join pipeline for production — checklist.▶
💡 Solution
# PRODUCTION JOIN OPTIMIZATION CHECKLIST: # 1. Filter BEFORE join (reduce data early) emp_filtered = emp_df.filter(F.col("salary") > 60000) # before join! result = emp_filtered.join(dept_df, "dept_id", "left") # 2. Broadcast small tables result = emp_df.join(broadcast(dept_df), "dept_id") # 3. Select only needed columns before joining emp_slim = emp_df.select("emp_id", "name", "dept_id", "salary") dept_slim = dept_df.select("dept_id", "dept_name") result = emp_slim.join(dept_slim, "dept_id") # 4. Use string key in join (avoids duplicate columns) result = emp_df.join(dept_df, on="dept_id", how="left") # 5. Enable AQE (Spark 3.x) — auto-optimizes at runtime spark.conf.set("spark.sql.adaptive.enabled", "true") spark.conf.set("spark.sql.adaptive.skewJoin.enabled", "true") # 6. Check execution plan result.explain("formatted") # look for: BroadcastHashJoin, SortMergeJoin # 7. Repartition on join key before repeated joins emp_repartitioned = emp_df.repartition(8, "dept_id") result = emp_repartitioned.join(dept_df, "dept_id") # less shuffle!
3B

Window Functions — ROW_NUMBER, RANK, DENSE_RANK, LAG, LEAD, NTILE, Running Totals

Row-level analytics without collapsing data. Ranking, running totals, comparing with previous rows — Data Engineering mein har jagah use hota hai.

Window.partitionBy Window.orderBy ROW_NUMBER RANK / DENSE_RANK LAG / LEAD NTILE SUM OVER FIRST_VALUE / LAST_VALUE ROWS BETWEEN

Key Concepts — Window Functions

  • Window spec — defines partition + order + frame. Created with Window class.
  • partitionBy — Group rows (like GROUP BY but keeps all rows).
  • orderBy — Order within each partition.
  • ROW_NUMBER() — Unique sequential number per partition. No ties.
  • RANK() — Same rank for ties, skips next rank (1,2,2,4).
  • DENSE_RANK() — Same rank for ties, no skip (1,2,2,3).
  • LAG(col, n) — Value from n rows BEFORE current row.
  • LEAD(col, n) — Value from n rows AFTER current row.
  • ROWS BETWEEN — Define sliding frame for aggregations.
  • Cannot use in WHERE — Must wrap in subquery/CTE to filter on window result.

02Why can't window functions be used in filter()? How to work around it?▶
💡 Solution
# Window functions execute AFTER filter in plan # So this FAILS: w = Window.orderBy(F.col("salary").desc()) # emp_df.filter(F.row_number().over(w) == 1) # ERROR! # WORKAROUND 1: withColumn then filter df = emp_df.withColumn("rn", F.row_number().over(w)) result = df.filter(F.col("rn") == 1).drop("rn") result.show() # WORKAROUND 2: Create temp view + SQL emp_df.createOrReplaceTempView("emp") spark.sql(""" SELECT * FROM ( SELECT *, ROW_NUMBER() OVER (ORDER BY salary DESC) rn FROM emp ) WHERE rn = 1 """).show()
03ROWS BETWEEN vs RANGE BETWEEN — explain with example.▶
💡 Solution
# ROWS BETWEEN: physical row count (position-based) w_rows = Window.partitionBy("emp_id").orderBy("month") \ .rowsBetween(-2, Window.currentRow) # always 3 rows max # RANGE BETWEEN: value-based range (logical) w_range = Window.partitionBy("emp_id").orderBy("revenue") \ .rangeBetween(-5000, Window.currentRow) # rows within ±5000 df = ms_df \ .withColumn("rows_avg", F.avg("revenue").over(w_rows)) \ .withColumn("range_avg", F.avg("revenue").over(w_range)) df.select("emp_id","month","revenue","rows_avg","range_avg").show() # Key difference: # ROWS: "previous 2 physical rows" — always same count # RANGE: "rows where value is within range" — count varies # Use ROWS for time-series windows (n previous periods) # Use RANGE for value-based windows (price range, score range)

Filter BEFORE window — reduce data first filtered = ms_df.filter(F.col("revenue") > 5000) w = Window.partitionBy("emp_id").orderBy("month") filtered.withColumn("rn", F.row_number().over(w)).show() # 3.
