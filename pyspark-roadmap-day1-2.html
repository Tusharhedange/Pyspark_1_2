SparkSession, SparkContext, RDD vs DataFrame, Schema & Data Types

PySpark ka foundation. Yahan se samjho ki Spark kaise kaam karta hai — entry point kya hai, RDD aur DataFrame mein kya fark hai, aur data ko schema ke saath kaise load karte hain.

SparkSession SparkContext RDD DataFrame StructType StructField Data Types Lazy Evaluation Transformations vs Actions
📖 Core Concepts

Key Things to Remember

  • SparkSession — Entry point for DataFrame/SQL API (Spark 2.0+). SparkContext andar hota hai.
  • SparkContext — Low-level entry point, RDD operations ke liye. sc = spark.sparkContext
  • RDD — Resilient Distributed Dataset. Low-level, no schema, Python objects. Slow on Python.
  • DataFrame — Schema + distributed table. Optimized via Catalyst optimizer. Always prefer over RDD.
  • Transformation — Lazy operation, returns new RDD/DF. e.g. filter(), select(), map()
  • Action — Triggers execution, returns result. e.g. show(), count(), collect(), write()
  • DAG — Directed Acyclic Graph. Spark execution plan banata hai transformations se.
  • StructType — Schema definition class. StructField(name, datatype, nullable) se banata hai.

01 Explain DAG and lazy evaluation — demonstrate with explain(). ▶
💡 Solution
# Lazy evaluation: transformations don't run immediately
# They build a DAG (execution plan)

df = emp_df \
    .filter(F.col("salary") > 70000) \
    .select("name", "salary") \
    .orderBy("salary", ascending=False)

# Nothing executed yet! Just builds plan.

# explain() shows the execution plan (DAG)
df.explain()           # parsed plan
df.explain(True)       # parsed + analyzed + optimized + physical
df.explain("extended") # full details

# Action triggers actual execution:
df.show()  # NOW it runs!

# Benefits of lazy eval:
# 1. Optimizer can reorder/merge operations
# 2. Predicate pushdown (filter early)
# 3. Only reads needed columns (projection pushdown)

01 What is the difference between SparkContext, SparkSession, and SQLContext? ▶
💡 Solution
# SparkContext (SC) — Spark 1.x era # - Entry point for RDD API # - One per JVM # - Low-level cluster communication sc = spark.sparkContext # SQLContext — Spark 1.x SQL API # - Entry for DataFrame/SQL # Deprecated in Spark 2.0 # SparkSession (Spark 2.0+) — UNIFIED entry point # - Combines SC + SQLContext + HiveContext # - One per application (singleton) # - Manages all spark functionalities spark = SparkSession.builder \ .appName("App") \ .enableHiveSupport() \ # optional: Hive metastore .getOrCreate() # Access underlying SC from session: sc = spark.sparkContext # Interview tip: Always use SparkSession. # SparkContext still needed for RDD operations only.
02 Explain Transformation vs Action — give 5 examples of each. ▶
💡 Solution
# TRANSFORMATIONS: lazy, return new DF/RDD, build DAG # Narrow (no shuffle): filter, select, withColumn, map, flatMap # Wide (requires shuffle): groupBy, join, orderBy, repartition, distinct # Transformation examples (nothing runs yet!) t1 = emp_df.filter(F.col("salary") > 70000) # narrow t2 = emp_df.select("name", "salary") # narrow t3 = emp_df.withColumn("bonus", F.col("salary") * 0.1) # narrow t4 = emp_df.groupBy("dept").agg(F.avg("salary")) # wide t5 = emp_df.join(dept_df, "dept") # wide # ACTIONS: eager, trigger execution, return result/write a1 = emp_df.count() # returns int a2 = emp_df.show() # prints to console a3 = emp_df.collect() # returns List[Row] a4 = emp_df.write.parquet("out/") # writes to storage a5 = emp_df.first() # returns first Row
03 What is a Catalyst Optimizer and how does it work? ▶
💡 Solution
# Catalyst Optimizer — Spark's query optimization engine # 4 phases: # 1. Analysis: Resolve column names, types → Logical Plan # 2. Logical Optimization: Apply rules (filter pushdown, constant folding) # 3. Physical Planning: Pick best join strategy, scan method # 4. Code Generation: Compile to JVM bytecode (Tungsten) # See the full plan: df = emp_df.filter(F.col("salary") > 70000).select("name", "salary") df.explain("extended") # Optimizations Catalyst does automatically: # - Predicate pushdown: filter before join # - Column pruning: read only needed columns # - Constant folding: 1+1 → 2 at compile time # - Join reordering: smaller table first # Example: filter pushed through join automatically result = emp_df.join(dept_df, "dept").filter(F.col("salary") > 80000) result.explain() # filter happens BEFORE join in physical plan!
04 Why should you always define schema explicitly in production? ▶
💡 Solution
# Problems with inferSchema=true in production: # 1. PERFORMANCE: Spark reads entire file TWICE (once for schema, once for data) df_bad = spark.read.option("inferSchema", "true").csv("huge_file.csv") # → Very slow on TB-scale data # 2. WRONG TYPES: "123" inferred as Int might break with "N/A" # 3. INCONSISTENCY: Schema can change between runs # SOLUTION: Always define schema explicitly from pyspark.sql.types import * schema = StructType([ StructField("emp_id", IntegerType(), False), StructField("name", StringType(), True), StructField("salary", DoubleType(), True), StructField("hire_date",DateType(), True), ]) df_good = spark.read.schema(schema).option("header", "true").csv("file.csv") # Benefits: # - Fast (no extra scan) # - Predictable types # - Fails fast on schema mismatch (good for data quality)
05 What is the difference between narrow and wide transformations? ▶
💡 Solution
# NARROW TRANSFORMATION: each partition depends on 1 parent partition # No shuffle needed! Fast, pipeline-able # Examples: map, filter, select, withColumn, flatMap, union narrow = emp_df.filter(F.col("salary") > 70000) # Each partition filtered independently # WIDE TRANSFORMATION: partition depends on MULTIPLE parent partitions # Requires SHUFFLE (data moves across network) — expensive! # Examples: groupBy, join, distinct, orderBy, repartition, reduceByKey wide = emp_df.groupBy("dept").count() # All "Engineering" rows collected from all partitions → new partition # Why it matters: # - Wide transformations = stage boundaries in DAG # - Each stage boundary = shuffle = disk I/O + network # - Minimize wide transformations for performance # - Filter BEFORE groupBy/join (reduce shuffle data) emp_df \ .filter(F.col("salary") > 70000) \ # narrow first! .groupBy("dept") \ # then wide .count() \ .show()
06 How does Spark handle NULL values differently from SQL? ▶
💡 Solution
# NULL behavior same as SQL in most cases, but some differences: # 1. Checking NULL — use isNull/isNotNull (not == None) emp_df.filter(F.col("salary").isNull()).show() emp_df.filter(F.col("salary").isNotNull()).show() # WRONG — returns empty! emp_df.filter(F.col("salary") == None) # 2. NULL in aggregations — ignored by default (like SQL) emp_df.select(F.avg("salary")).show() # ignores NULL Meena # 3. eqNullSafe — NULL-safe equality emp_df.filter(F.col("salary").eqNullSafe(None)).show() # 4. NULL in sort — placed at end by default emp_df.orderBy(F.col("salary").asc_nulls_first()).show() emp_df.orderBy(F.col("salary").desc_nulls_last()).show() # 5. coalesce — first non-null value emp_df.select(F.coalesce(F.col("salary"), F.lit(0))).show()
07 What is a Spark Stage and how is it related to shuffle? ▶
💡 Solution
# JOB → STAGES → TASKS (hierarchy) # # JOB: Created per Action (show, count, write) # STAGE: Group of narrow transformations between shuffles # TASK: One unit of work per partition per stage # # Example: # emp_df.filter().select().groupBy().count().show() # → Job 1 # → Stage 1: filter + select (narrow, no shuffle) # → SHUFFLE (groupBy boundary) # → Stage 2: aggregate + count (after shuffle) # # More stages = more shuffles = slower # View in Spark UI: localhost:4040 # Jobs → click job → see stages + tasks # Avoid extra stages: emp_df \ .filter(F.col("salary") > 70000) \ # Stage 1 .select("name", "dept", "salary") \ # Stage 1 (same) .groupBy("dept") \ # Shuffle boundary .agg(F.avg("salary")) \ # Stage 2 .show() # Action → triggers job

02

DataFrame Operations — select, filter, withColumn, drop, rename, groupBy, agg

DataFrame manipulations jo roz kaam aate hain. Data transform karna, columns add/remove karna, aggregate karna — ye sab Data Engineering ka core hai.

select / selectExpr filter / where withColumn drop / rename groupBy + agg orderBy / sort distinct / dropDuplicates alias / col / lit when / otherwise

Key Things to Remember

  • col() vs F.col() — Same thing. F is alias for pyspark.sql.functions. Use F.col() always.
  • df["col"] vs F.col("col") — Both work, but F.col() is preferred for clarity.
  • withColumn — Adds new column OR overwrites existing. Returns new DF (immutable!).
  • filter vs where — Identical. Just aliases of each other.
  • select vs selectExpr — selectExpr allows SQL expressions as strings.
  • agg() — Used with groupBy. Can apply multiple aggregations at once.
  • when/otherwise — PySpark's CASE WHEN equivalent.
  • lit() — Create a column with a literal/constant value.

# collect_list: aggregates all values into array (with duplicates) emp_df.groupBy("dept").agg( F.collect_list("name").alias("all_employees") ).show(truncate=False) # collect_set: unique values only sales_df.groupBy("emp_id").agg( F.collect_set("category").alias("categories_sold") ).show(truncate=False) # OOM RISK: If one group has millions of rows: # collect_list → single array with millions of items → driver OOM! # Safer alternatives: # 1. Limit array size: F.slice(F.collect_list("name"), 1, 100) # 2. Sample first, then collect # 3. Use concat_ws for string joining instead # concat_ws alternative (memory efficient): emp_df.groupBy("dept").agg( F.concat_ws(", ", F.collect_list("name")).alias("emp_csv") ).show(truncate=False)

10 Full ETL pipeline: Read → Clean → Transform → Validate → Write. ▶
💡 Solution
from pyspark.sql import SparkSession, functions as F from pyspark.sql.types import * spark = SparkSession.builder.appName("ETLPipeline").getOrCreate() # ── STEP 1: READ ── schema = StructType([ StructField("emp_id", IntegerType(), False), StructField("name", StringType(), True), StructField("dept", StringType(), True), StructField("salary", DoubleType(), True), ]) raw_df = spark.createDataFrame(emp_data, schema) # ── STEP 2: CLEAN ── clean_df = raw_df \ .dropDuplicates(["emp_id"]) \ .withColumn("name", F.trim(F.upper(F.col("name")))) \ .withColumn("dept", F.trim(F.col("dept"))) \ .fillna({"salary": 0.0, "dept": "UNKNOWN"}) # ── STEP 3: TRANSFORM ── transformed_df = clean_df \ .withColumn("salary_band", F.when(F.col("salary") > 80000, "SENIOR") .when(F.col("salary") > 60000, "MID") .otherwise("JUNIOR")) \ .withColumn("annual_bonus", F.col("salary") * 0.10) \ .withColumn("load_ts", F.current_timestamp()) \ .withColumn("source", F.lit("HR_SYSTEM")) # ── STEP 4: VALIDATE ── valid = transformed_df.filter(F.col("emp_id").isNotNull()) invalid = transformed_df.filter(F.col("emp_id").isNull()) print(f"Valid: {valid.count()} | Invalid: {invalid.count()}") # ── STEP 5: WRITE ── valid.write \ .mode("overwrite") \ .partitionBy("dept") \ .parquet("silver/employees/") print("Pipeline complete!")
